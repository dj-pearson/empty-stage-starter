## Codebase Patterns
- Edge functions live in `functions/<function-name>/index.ts` (Supabase Edge Functions use Deno)
- Edge function config goes in `supabase/config.toml` under `[functions.<name>]` with `verify_jwt` setting
- Use `https://deno.land/std@0.168.0/http/server.ts` serve() and `https://esm.sh/@supabase/supabase-js@2` createClient()
- Shared edge function utilities in `functions/_shared/`: cors.ts (CORS headers), auth.ts (JWT verification), url-validator.ts (SSRF prevention)
- All edge functions use `getCorsHeaders(req)` from `../_shared/cors.ts` - no more wildcard CORS
- All privileged edge functions use `authenticateRequest(req)` from `../_shared/auth.ts` for JWT verification
- `functions/sitemap.xml.ts` is a Cloudflare Pages function, NOT a Supabase edge function
- Edge function error responses must return generic messages to clients (`'Internal server error'`), log details server-side with `console.error`
- Always derive user identity from JWT (supabase.auth.getUser()), never trust client-supplied user_id
- Template patterns available in `edge-functions-template/functions/` for reference
- TypeScript check: `npx tsc --noEmit`, Tests: `npx vitest run` (382 passing)
- Test files follow pattern: `src/**/*.test.{ts,tsx}` alongside source files
- Mock Supabase client in tests by mocking `@/integrations/supabase/client`
- Auth page tests need AppProvider wrapper and mocks for: logger.withContext, login-history, use-toast, react-router-dom (useSearchParams), platform
- Schema components: ArticleSchema/FAQSchema/BreadcrumbSchema use Helmet (mock react-helmet-async); RecipeSchema uses useEffect + DOM manipulation (test via document.getElementById)
- US-001 through US-006 already completed strict mode, dead code cleanup, env validation, error boundaries, route error boundaries, and API retry logic
- Build: `npx vite build` (the `npm run build` has an Expo prebuild step that requires native deps; Cloudflare Pages should use `npx vite build`)

- `Tips.Event` instances must be stable across app launches — declaring them as `static let` on an enum namespace (TipEvents) is the idiomatic approach.
- `#Rule(event) { $0.donations.count == 0 }` is the cleanest "show until used once" predicate; no need to track "shown before" state manually.
- `.popoverTip(tip)` anchors the popover to the view it's attached to; put it on a persistent element near the rows, not on the rows themselves (rows may scroll off screen and the tip would get cut off).
- Donating the event from inside a SwiftUI Button closure requires `Task { await event.donate() }` — `donate()` is async.

- Log: "Log `<slot>` as `<result>` in EatPal", "Mark `<slot>` `<result>` in EatPal", "Record `<slot>` in EatPal".

### Design decisions
- Background execution (`openAppWhenRun = false`) is the whole point — if the user wanted to open the app, they would. Siri fires the intent, the result dialog appears in the Siri UI overlay, and nothing forces the user out of their current context.
- All intent writes use `addedVia: "siri"` so analytics and ReasoningBank can distinguish Siri-sourced data.
- No kid-entity picker in this pass. Today's meal log applies to every kid with a matching slot on today's date. Follow-up: donate per-kid LogMealResultIntent variants so Siri learns per-kid patterns.
- No donation from manual flows yet. AppShortcutsProvider-declared phrases already work on iOS 16+ without donation; donation only improves Siri suggestions. Intentionally deferred.

### Not verified yet
- Requires `xcodegen generate` + Xcode build to compile the AppIntents metadata. Siri registration happens at first launch after install.
- On-device test: "Hey Siri, add milk to EatPal grocery" is the canonical smoke test.
- Shortcuts app discovery: the three tiles should appear under EatPal automatically after first launch.

### Learnings
- `AppShortcutsProvider` replaces the old INInteraction donation pattern for surface-level Siri discovery — the system parses your phrases at install time.
- `(.applicationName)` in phrase strings is iOS 16+ and keeps phrases in sync with any future app rename.
- `@Parameter` with `inputOptions: .init(capitalizationType: .sentences)` makes Siri transcription feel natural in dialogs — without it the dictated word can be all-lowercase.
- `DataService.shared` being `@MainActor` + having `ensureUserId` built in means App Intents need zero extra auth plumbing — the keychain Supabase session gets used automatically.
- MealSlot → MealSlotAppEnum mapping: keeping Siri vocabulary lean (4 options) and mapping snack → {snack1, snack2} internally is friendlier than forcing users to say "snack one".

## 2026-04-17 - US-150: OfflineStore durable write queue + NetworkMonitor wiring (DONE first pass)

### What was implemented
Promoted OfflineStore.swift from a sync-delete placeholder into a durable queue that replays insert / update / delete mutations against Supabase when the device regains connectivity. NetworkMonitor now triggers the drain automatically on offline→online transitions. AppState's grocery mutations route to the queue instead of rolling back when the failure is network-related.

### Files changed
- `ios/EatPal/EatPal/Services/OfflineStore.swift`
  - Added `Op` enum (insert/update/delete) and `Table` enum (foods, kids, recipes, plan_entries, grocery_items).
  - New `@Published` state: `pendingMutationCount`, `isSyncing`, `lastSyncError` for UI diagnostics.
  - New typed enqueue helpers: `enqueueInsert<T: Encodable>`, `enqueueUpdate<T: Encodable>`, `enqueueDelete(table:entityId:)`.
  - `syncPendingMutations()` rewritten with concurrent-run guard, per-table dispatch (groceryItems / foods / planEntries / kids / recipes) that decodes stored JSON payloads back into domain structs and posts the right typed insert/update/delete to Supabase.
  - Sync failures route through SentryService.capture with a `queue: "OfflineStore"` tag; sync stops on first failure to preserve ordering.
  - `refreshPendingCount()` fires after every enqueue / clear so the published count stays accurate.

- `ios/EatPal/EatPal/Utilities/NetworkMonitor.swift`
  - Tracks offline→online edge. On the transition, leaves a Sentry breadcrumb and awaits `OfflineStore.shared.syncPendingMutations()`.

- `ios/EatPal/EatPal/App/AppState.swift`
  - `addGroceryItem`, `deleteGroceryItem`, `toggleGroceryItem` updated with an `if isNetworkError(error) { ... }` branch that keeps the optimistic state AND queues the mutation, plus an info-level "Queued for sync" toast on add.
  - New `isNetworkError(_:)` helper inspects URLError / NSURLErrorDomain codes (notConnectedToInternet, networkConnectionLost, timedOut, cannotFindHost, cannotConnectToHost, internationalRoamingOff, dataNotAllowed) to distinguish connectivity failures from semantic failures.

### Behaviour summary
- User adds "milk" to grocery while offline → row appears in the list, toast says "Queued for sync", OfflineStore has 1 pending mutation. User comes back online → NetworkMonitor fires, OfflineStore drains the queue, Supabase gets the insert.
- User deletes an item offline → row disappears locally, delete queued with table=grocery_items, operation=delete.
- Non-network errors (validation, RLS reject) still rollback the optimistic state — queueing is reserved for transient connectivity failures.

### Not in this pass (follow-ups)
- Remaining AppState methods (updateGroceryItem, addFood/updateFood/deleteFood, plan_entries, kids, recipes) still follow the classic optimistic-then-rollback pattern. Retrofitting them is mechanical but intentionally scoped out to keep this change focused.
- No in-app queue diagnostic view yet (Settings row showing pendingMutationCount + lastSyncError is a good next-pass addition).
- No conflict resolution against the realtime channel — last-write-wins in practice, which is fine for grocery items but may need a rethink for plan entries where two devices could vote differently.
- JSON encoding uses the default JSONEncoder (matching domain-struct CodingKeys); the private `JSONEncoder.supabaseSnakeCase` extension is a placeholder in case we switch to `.convertToSnakeCase` in the future.

### Learnings
- `URLError` and `NSURLErrorDomain` overlap but aren't identical — covering both code paths in `isNetworkError` handles the various stacks in supabase-swift / URLSession / async-http-client.
- Preserving queue ordering by stopping on the first failure is important for correctness (a delete followed by an insert of the same id should NOT be reordered).
- SwiftData-backed queue persists across app launches, which is necessary — a user might kill the app between going offline and coming back online.

## 2026-04-17 - US-152: Widget freshness via App Group UserDefaults (DONE)

### What was implemented
Home-screen widget data now refreshes every time the relevant AppState collections mutate, via a debounced Combine pipeline that writes a snapshot to the shared App Group UserDefaults and asks WidgetKit to reload timelines.

### Files changed / added
- `ios/EatPal/EatPal/Services/WidgetSnapshot.swift` (new, ~90 LOC)
  - `@MainActor enum WidgetSnapshot` namespace: `appGroup = "group.com.eatpal.app"`, `Key` constants (meals, grocery_count, pantry_low_count, tonight_dish, try_bite_streak, last_updated_at), `widgetKind = "EatPalMealWidget"`.
  - `Payload` struct holds the snapshot (meals array with slot/foodName/icon, grocery / pantry-low / try-bite counts, tonight's dish name).
  - `write(_:)` debounces 0.5s via DispatchWorkItem; `writeImmediately(_:)` bypasses for launch / background-handoff paths.
  - `persist(...)` writes each key and calls `WidgetCenter.shared.reloadTimelines(ofKind:)`.

- `ios/EatPal/EatPal/App/AppState.swift`
  - `init()` now calls `setupWidgetSnapshotSync()`.
  - Combine sink: `$planEntries.combineLatest($foods, $groceryItems, $recipes).debounce(500ms).sink { refreshWidgetSnapshot() }` stored in a new `cancellables: Set<AnyCancellable>`.
  - `refreshWidgetSnapshot()` composes the Payload: today's meals for the active kid (or across all kids), tonight's dinner name (recipe preferred over food), pantry-low count (qty ∈ (0, 2]), unchecked grocery count, try-bite streak placeholder.

- `ios/EatPal/EatPal/EatPalWidget/EatPalWidget.swift`
  - `MealPlanEntry` extended with `pantryLowCount`, `tonightDish: String?`, `tryBiteStreak`.
  - `loadFromAppGroup()` reads the three new keys.
  - Placeholder + #Preview updated to include the new fields.

### Design decisions
- Combine observation of @Published collections means the snapshot auto-refreshes for every mutation path — the app never has to call `WidgetSnapshot.write(...)` manually, and we can't forget.
- 500ms debounce avoids thrashing WidgetKit during batch inserts (recipe→grocery, scan-a-list, voice-add multi-item).
- Payload uses primitive-only dict for `meals` so reading from the widget doesn't require access to the domain Codable types — matches the existing key naming ("widget_meals") so the migration is zero-downtime.
- try-bite streak is a 0 placeholder; the actual streak calc isn't scoped here but the storage path is ready.

### Not in this pass
- Lock Screen widgets (`.accessoryRectangular`, `.accessoryInline`, `.accessoryCircular`) — covered by US-146 next.
- Interactive widget buttons (check off a grocery item from the home screen) — covered by US-147.
- Using `AppGroup` for the OfflineStore SwiftData container so the widget target can also read cached data — overkill; App Group UserDefaults already covers the widget's needs.

### Learnings
- Widgets live in their own process. The only realistic IPC surfaces between app ↔ widget are App Group UserDefaults (for small snapshots), shared files (for bigger blobs), and Core/SwiftData with an App-Group-scoped store. UserDefaults is the right tool for the widget data here — small, flat, flush-on-write.
- `WidgetCenter.shared.reloadTimelines(ofKind:)` takes the `kind` string that matches the `StaticConfiguration`'s kind. Storing it in `WidgetSnapshot.widgetKind` avoids magic-string drift.
- Combine's `combineLatest` fires once for every upstream publisher change. Debouncing by 500ms smooths bursts cleanly; no need for `.removeDuplicates` since the snapshot contents are what the widget ultimately cares about.

## 2026-04-17 - US-146: Lock Screen widget families (accessoryRectangular / Inline / Circular) (DONE)

### What was implemented
Extended the existing EatPalWidget with three Lock Screen families so users can see tonight's dinner, grocery count, pantry status, or their try-bite streak at a glance from the Lock Screen.

### Files changed
- `ios/EatPal/EatPalWidget/EatPalWidget.swift`
  - New view structs:
    - `MealPlanAccessoryRectangular` — two-line layout with "Tonight" header + dish name + grocery/pantry status chips. Uses `.widgetAccentable()` so icons and the "Tonight" label pick up the OS tint.
    - `MealPlanAccessoryInline` — single tinted Label using iOS's inline widget style. Priority order: tonight's dish > grocery count > pantry-low count > EatPal branding fallback.
    - `MealPlanAccessoryCircular` — ZStack with `AccessoryWidgetBackground()` + stacked icon/number. Shows the try-bite streak when > 0, otherwise grocery count.
  - New `MealPlanWidgetView` router: reads `@Environment(\.widgetFamily)` and dispatches to the right sub-view. Handles `.containerBackground(for: .widget)` with `.systemBackground` for the rectangular system widgets and `Color.clear` for the accessory widgets (Lock Screen widgets get the system's tinted mask automatically).
  - `EatPalMealWidget.supportedFamilies` extended from `[.systemSmall, .systemMedium]` to include `.accessoryRectangular`, `.accessoryInline`, `.accessoryCircular`.
  - `configurationDisplayName` / `description` tightened to reflect all the data surfaced.

### Design decisions
- **Rectangular** is the most information-dense; it should be the user's default pick. Led with tonight's dish because that's the single most useful piece of "glanceable meal planning."
- **Inline** is space-constrained — I prioritised tonight's dish because the user has already explicitly chosen to show EatPal on their Lock Screen; showing "EatPal" alone would be useless.
- **Circular** favours the try-bite streak because it's a repeat-delight loop (streak go up = motivation) and falls back to grocery count when there's no streak to show.
- All three read from the same App Group snapshot the home-screen widgets already use (US-152) — no new writes needed.
- `.widgetAccentable()` is the right tool for accessory widgets to opt in to OS-tinted rendering. Text inside is dimmed by default; accentable elements are shown at full contrast.

### Not in this pass
- Interactive widget buttons (tap to check off grocery items) — scoped to US-147 because it needs an AppIntent that can execute from the widget extension process.
- Control Center widget (US-148) is iOS 18+ and a separate target class.
- Deep links from widgets — tapping a Lock Screen rectangular widget should open the Grocery or Planner tab directly. The `DeepLinkHandler` already understands eatpal://openGroceryList / openMealPlan; wiring `widgetURL(_:)` onto each view is a ~5-line follow-up.

### Learnings
- Accessory widget container background is special: pass `Color.clear` (or skip `containerBackground` — the system applies its own tinted mask). Passing a solid color breaks the Lock Screen look.
- `AccessoryWidgetBackground()` is a ready-made circular background that matches Apple's own widgets. Use it on circular.
- `widgetAccentable()` takes no argument; it just marks the view as eligible to be drawn in the accent tint. No-op on system-size widgets.
- `MealPlanWidgetView` router pattern is cleaner than per-family `#if`/`#available` blocks inside the widget configuration — keeps each family's layout in its own struct.

## 2026-04-17 - US-144: HealthKit nutrition write-back on meal log (DONE)

### What was implemented
Users who opt in now have their logged meals' nutrition written to the iPhone Health app. When a plan entry transitions to `ate` and the linked recipe has nutrition data, EatPal bundles the macros into a single HKCorrelation of type `.food` so Health surfaces a coherent meal entry rather than four disconnected quantity samples.

### Files changed / added
- `ios/EatPal/EatPal/EatPal.entitlements`
  - Added `com.apple.developer.healthkit` (true) and the empty `com.apple.developer.healthkit.access` array (defaults to standard dietary access).

- `ios/EatPal/EatPal/Resources/Info.plist`
  - Added NSHealthShareUsageDescription (opt-in wording around reading) and NSHealthUpdateUsageDescription (opt-in wording around writing) — both explicit about the "only if you opt in" nature.

- `ios/EatPal/EatPal/Services/HealthKitService.swift` (new, ~130 LOC)
  - `@MainActor final class HealthKitService` singleton.
  - `isAvailable` proxies HKHealthStore.isHealthDataAvailable.
  - `isEnabled` is UserDefaults-backed (`eatpal_health_sync_enabled`) — distinct from the system-level HealthKit authorization; users can toggle off in-app without revoking system access.
  - `requestAuthorization()` asks for share (not read) on the four macros + the `.food` correlation type. Returns true if at least one type reports `.sharingAuthorized` after the prompt.
  - `writeMeal(calories:proteinGrams:carbsGrams:fatGrams:mealDate:mealName:)` no-ops when `!isEnabled` or `!isAvailable`; otherwise constructs one HKQuantitySample per non-nil/non-zero macro and bundles them into a single `HKCorrelation(type: .food, start: end:, objects:, metadata:)` with `HKMetadataKeyFoodType` set to the meal name. No kid names, no EatPal IDs — HealthKit is not our PII bucket.

- `ios/EatPal/EatPal/App/AppState.swift`
  - `updatePlanEntry` now fires `Task { await writeHealthSample(for: planEntries[index]) }` when `updates.result == MealResult.ate.rawValue`.
  - New private `writeHealthSample(for:)` resolves the linked recipe + nutritionInfo, formats the plan's date string back to `Date`, and calls the service. SentryService breadcrumb on success, capture on error.

- `ios/EatPal/EatPal/Views/Settings/HealthSettingsView.swift` (new, ~95 LOC)
  - Form with one Toggle that triggers `requestAuthorization` on flip-to-on.
  - Denial alert with a "Open Settings" button linking to the iOS Settings app.
  - Link to the Health app via `x-apple-health://`.
  - Footer copy adapts to current state (enabled / disabled / unavailable) so users always know what the toggle means.

- `ios/EatPal/EatPal/Views/Settings/SettingsView.swift`
  - Preferences section gained a "Sync to Health" NavigationLink between "Appearance" and the existing support links.

### Design decisions
- Opt-in is a two-gate affair: the app-level toggle (`isEnabled`) AND HealthKit's system authorization. Toggle flips are the single source of truth for our side; system auth is owned by iOS.
- HKCorrelation over individual samples — meals in Health should read as meals, not as a scattered timeline of macro points. HealthKit's own Food tab joins them the same way.
- Only recipes carry nutrition today; the Food model has no macros field. Food-only plan entries are skipped silently — there's no honest number to write. Adding macros to Food is a data-model change that belongs to a separate story.
- No reads, no imports: this pass is write-only. Reading HealthKit data would open a consent surface the user doesn't need yet.

### Not in this pass
- Importing dietary data FROM Health for the AICoach / meal planning suggestions.
- Per-kid correlation — HealthKit is anchored to the device owner's identity; multi-kid household tracking in Health isn't a supported pattern and would surprise users. Keeping the correlation anonymous-ish (just food name metadata) is safer.
- A "Last synced N minutes ago" status line in the Settings screen — simple follow-up.

### Learnings
- `HKHealthStore.requestAuthorization(toShare:read:)` doesn't report whether the user tapped Allow or Deny (Apple privacy design). The post-request check is to look at `authorizationStatus(for:)` per type and infer intent.
- `HKCorrelation` needs the contained samples to all be authorised; if one type is denied, the whole correlation fails to save. Worth surfacing as a single generic error to the user rather than per-type diagnostics.
- `HKMetadataKeyFoodType` is the right slot for a human-readable meal name; the system shows it in the detail view in Health.
- `x-apple-health://` opens the Health app to its main tab — useful for a "verify your data" shortcut from Settings.

## 2026-04-17 - US-145: Live Activities + Dynamic Island (grocery trip) (DONE first pass)

### What was implemented
Users can start a "shopping trip" Live Activity that tracks progress through the grocery list on the Lock Screen and in the Dynamic Island. Every time an item is checked off (online or offline-queued), the activity updates with the new count and the name of the just-added item. Trip auto-ends 3 seconds after hitting 100%.

Scoped down from the original US-145 multi-activity brief: cook-mode Live Activity is deferred until iOS has a CookMode view (web has one; iOS doesn't yet). AI-meal-plan-generation activity isn't worth the ceremony because the flow is short (seconds, not minutes).

### Files changed / added
- `ios/EatPal/EatPal/Resources/Info.plist`
  - `NSSupportsLiveActivities = true`
  - `NSSupportsLiveActivitiesFrequentUpdates = true` (lets us update on every check-off without being throttled)

- `ios/EatPal/project.yml`
  - Created a new source folder reference `path: Shared` listed under BOTH the `EatPal` target and the `EatPalWidgetExtension` target. XcodeGen compiles the shared Swift files into both binaries — the clean way to share ActivityAttributes between main app and widget.

- `ios/EatPal/Shared/GroceryTripActivityAttributes.swift` (new, ~35 LOC)
  - `public struct GroceryTripAttributes: ActivityAttributes`.
  - `TripState: Codable, Hashable` — totalCount, checkedCount, lastCheckedName, with computed `progress` and `isComplete`.
  - Static attrs: `listTitle`, `startedAt`.
  - `public` API surface because the widget extension lives in a different module.

- `ios/EatPal/EatPal/Services/GroceryTripActivityService.swift` (new, ~95 LOC)
  - `@MainActor` singleton. One active activity at a time; init recovers any in-flight Activity from a previous launch.
  - `isAvailable` checks `ActivityAuthorizationInfo().areActivitiesEnabled`.
  - `start(listTitle:totalCount:checkedCount:)` constructs Attributes + TripState, requests an Activity with an 8-hour staleDate safety net.
  - `update(totalCount:checkedCount:lastCheckedName:)` coalesces state updates; auto-ends 3 seconds after `isComplete` becomes true so the user sees the 100% bar.
  - `end(finalState:dismissalPolicy:)` cleans up.
  - All failures route to SentryService.

- `ios/EatPal/EatPalWidget/GroceryTripLiveActivity.swift` (new, ~95 LOC)
  - `Widget` conforming to `ActivityConfiguration(for: GroceryTripAttributes.self)`.
  - Lock Screen view: title + progress bar + count + "Just added:" subtitle or "All done!" banner.
  - Dynamic Island expanded: leading (title), trailing (count), center (last added), bottom (progress bar).
  - Compact leading/trailing: cart icon + count ratio.
  - Minimal: cart icon.
  - `widgetURL(URL(string: "eatpal://openGroceryList"))` so tapping the activity deep-links into the grocery tab via DeepLinkHandler.

- `ios/EatPal/EatPalWidget/EatPalWidget.swift`
  - `EatPalWidgetBundle.body` now includes `GroceryTripLiveActivity()` alongside `EatPalMealWidget()`.

- `ios/EatPal/EatPal/App/AppState.swift`
  - `toggleGroceryItem` captures item name upfront and, on either success or offline-queue fallback, calls `updateGroceryTripActivity(lastCheckedName: checked ? name : "")`.
  - New private `updateGroceryTripActivity(lastCheckedName:)` recalculates totals and updates the activity — no-op when none is active.

- `ios/EatPal/EatPal/Views/Grocery/GroceryView.swift`
  - Quick-add menu gained a divider + Start shopping trip button (when none active, disabled if the list is empty) OR End shopping trip button (destructive-styled, when active). Start passes initial total/checked counts so the activity begins accurate.

### Design decisions
- **Shared attributes via a new Shared/ folder** rather than duplicating the struct or reaching for a Swift package. XcodeGen's multi-target `path: Shared` is the idiomatic fit.
- **One activity at a time**. Start-while-active becomes an update (the user fell out of the flow once; don't stack Lock Screen banners on them).
- **3-second delay before auto-end** after 100%: long enough for the win feel, short enough not to crowd the Lock Screen.
- **Offline updates work**: even when a toggle queues via OfflineStore, the Live Activity still updates locally. The user sees their trip progress regardless of network.
- **Dynamic Island compact trailing shows "checked/total"** — the single most useful glance in a crowded UI.

### Not in this pass
- Cook-mode Live Activity. Unblocked once iOS has a CookMode view.
- AI-meal-plan-generation activity. Waiting until the flow becomes slow enough to warrant one.
- Push-updated Live Activities (ActivityKit supports pushType:.token). Useful when another device / the web app toggles an item mid-trip; punt.
- "Shopping at <store>" label — would require the grocery_lists.storeName field surfaced in the UI; trivial follow-up.

### Learnings
- Shared types between app + widget: multi-target source paths in XcodeGen (`path: Shared` under both targets) is cleaner than a Swift package and more discoverable than file-by-file target membership juggling.
- `ActivityContent(state:staleDate:)` with a long staleDate (8 hours) is the right abandonment safety net — iOS will silently expire stuck activities without user action.
- `ActivityAuthorizationInfo().areActivitiesEnabled` respects both the system-wide Live Activities toggle and any per-app setting.
- `Activity<A>.activities.first` on init is the recovery pattern; without it, force-quit + relaunch loses the handle even though the activity keeps rendering on the Lock Screen.
- `.frequentUpdates` entitlement unlocks rapid update calls without throttling — required when updating on every checkoff.
